Phase 2

The Box Model & Sizing Units

The most critical phase — every element is a box. Master how boxes are sized, spaced, and measured before touching layout systems.

1 — The Box Model

Box Model — Anatomy Diagram

Every single HTML element — from a tiny <span> to a full-page <section> — is rendered as one of these boxes. The model has four layers from inside out: content → padding → border → margin.

width & height — Content Dimensions

width: 160px
height: 80px
width: 50%
height: 80px
width: auto
height: auto (content-driven)
max-width: 400px — This box will never grow beyond 400px even if the container is wider.
w

width: auto (default for block elements) means the element fills available space. height: auto means the element grows to fit its content. max-width is essential for readable text columns — set it to 60–75ch or 680px on article containers.

padding — Inside Spacing (pushes content inward)

padding: 24px all 4 sides
padding: 32px 8px top+bottom / left+right
padding: 8px 32px 24px top / left+right / bottom
padding: 8px 32px 24px 4px top / right / bottom / left

Memory trick for 4-value shorthand: TRBL ("trouble") — Top, Right, Bottom, Left. Goes clockwise from the top. Individual properties: padding-top, padding-right, padding-bottom, padding-left.

border — The Line Around Padding

border: 2px solid #5b8dee
border: 2px dashed #e07b54
border: 3px dotted #79d4b5
border: 6px solid #9099b0
border-bottom: 3px solid #5b8dee
Common for nav links, input underlines
border-radius: 4px
border-radius: 12px
border-radius: 999px (pill)
border-radius: 50%

border shorthand order: width style color. The style value is REQUIRED — without it no border renders even if width and color are set. Common styles: solid, dashed, dotted, none. border-radius: 50% on a square element = perfect circle.

margin — Outside Spacing (pushes other elements away)

margin: 0 auto Horizontally centered via auto left/right margins

⚠️ Margin Collapse — The Most Common Gotcha

margin-bottom: 40px
margin-top: 24px Gap between boxes = 40px (not 64px)

Negative Margin — Pull Elements Together

Base element
margin-top: -20px — overlaps the element above

Margin collapse: When two vertical margins meet, only the LARGER one wins (they don't add). This only happens on vertical (top/bottom) margins between block elements — never on flex or grid children, and never on horizontal margins.

margin: 0 auto only works on block elements with an explicit width set. It has no effect on inline elements.

2 — Box Sizing: content-box vs border-box

box-sizing — The Most Important CSS Rule

box-sizing: content-box (default ❌)

width: 300px
padding: 30px
Actual rendered width: 360px
(300 + 30 + 30 padding)

You declared width: 300px. You added padding: 30px. The browser renders it as 360px. Layout breaks.

box-sizing: border-box ✅

width: 300px
padding: 30px
Actual rendered width: 300px
(padding is absorbed inside)

You declared width: 300px. You added padding: 30px. The browser renders it as 300px. Layout is predictable. ✅

This is why every modern CSS reset begins with:
*, *::before, *::after { box-sizing: border-box; }
Apply it globally and never think about it again.

3 — Units of Measurement

px — Absolute Pixels

border: 1px ✅ px is great for borders
box-shadow: 0 4px 12px ✅ px for shadows / fine offsets
font-size: 14px ⚠️ Overrides user browser settings

px is an absolute unit — it does not scale with anything. Use for: borders, shadows, fixed-pixel constraints. Avoid for: font sizes, padding, and spacing — these should scale with user preferences.

rem — Root em (Relative to <html>)

0.25rem = 4px
0.5rem = 8px
0.75rem = 12px
1rem = 16px (root default)
1.25rem = 20px
1.5rem = 24px
2rem = 32px
3rem = 48px

This text uses font-size: 1rem and padding: 1rem 1.5rem. If a user increases their browser base font size from 16px to 20px, this entire component — text AND padding — scales up proportionally. 🎉

rem always looks at the <html> element's font-size. Since that's usually the browser/OS preference, rem automatically respects accessibility settings. Best practice: use rem for all font sizes, padding, margin, and gap values.

em — Relative to Parent font-size

Parent: font-size: 16px (1em = 16px)
Child: font-size: 1.25em → 20px
Grandchild: font-size: 1.25em → 25px
⚠️ Compounding! Each level multiplies the parent's size.

✅ Good use of em — component-relative padding

em is relative to the element's OWN font-size (for most properties) or its PARENT's font-size (for the font-size property itself). This causes "compounding" — nested ems multiply.

✅ Use em for: padding/margin INSIDE a component that should scale with the component's own font-size (e.g., button padding).
⚠️ Avoid em for font-size in nested structures — use rem instead for predictability.

vw / vh — Viewport Width / Height

width: 50vw 50% of browser window width — not parent container
height: 30vh 30% of browser window height
4vw font font-size: 4vw Scales with viewport — careful on mobile
height: 100vh — Full-screen hero section (classic usage)

100vw includes scrollbar width, which can cause horizontal overflow. Use width: 100% on block elements instead. 100vh is useful for full-screen hero sections and modals — but see the mobile gotcha below.

svh / lvh / dvh — Modern Mobile Viewport Units

svh

Small Viewport Height

Based on the smallest possible viewport — when browser navigation bars are fully visible. Content here is always visible. ✅

Use for: hero sections, full-screen landing pages

lvh

Large Viewport Height

Based on the largest possible viewport — when browser UI has scrolled away. May initially clip content behind the browser toolbar.

Use for: elements only shown after user scrolls

dvh

Dynamic Viewport Height

Continuously updates as the browser toolbar shows/hides. Can cause layout "jumps" and performance issues on scroll.

⚠️ Use sparingly — causes reflow on scroll

📱 Recommendation: Replace height: 100vh on mobile hero sections with height: 100svh. This prevents content from hiding behind the mobile browser's address bar / navigation bar.

The problem: on mobile, 100vh is calculated when the browser UI is hidden (large viewport). When the user first lands, the browser toolbar is visible, so the section is taller than the screen — the bottom of your hero is cut off. svh solves this.

📊 Unit Decision Matrix — Quick Reference

Unit Relative to Best for Avoid for
px Nothing (absolute) Borders, shadows, fine offsets Font sizes, padding (accessibility)
rem <html> font-size Font sizes, spacing, padding ✅ Component-internal scaling (use em)
em Parent font-size Component padding that scales with its own font Nested font-sizes (compounding risk)
vw Viewport width Full-width layouts, fluid type General sizing (scrollbar issues)
vh Viewport height Desktop full-height sections Mobile full-height (use svh instead)
