Phase 3

Document Flow & Positioning

How browsers decide where elements live — and how to move them out of that default flow using display, position, and z-index.

1 — The display Property & Normal Flow

What is Normal Flow?

Before any CSS positioning or layout tricks, browsers place elements according to Normal Flow — the default reading order. Block elements stack vertically (top to bottom). Inline elements flow horizontally (left to right) and wrap like words in a sentence. Everything in Phase 3 is about understanding and escaping this flow.

display: block

Block 1 — <div> Takes full width → forces a new line
Block 2 — <p> Stacks below regardless of content width
Block 3 — <section> Respects width, height, and all margins

Native block elements: <div>, <p>, <h1–h6>, <section>, <article>, <header>, <footer>, <ul>, <li>. You can force any element to block with display: block.

display: inline

This is a paragraph. The word span is inline — it sits inside the text flow. So is em, and strong, and links. They wrap naturally when the line runs out of space, just like these words do. Notice they cannot have width or height applied.

⚠️ Inline elements ignore width, height, and vertical margin-top / margin-bottom. Only horizontal padding/margin applies. If you need sizing, use inline-block instead.

display: inline-block

Tag w:80px h:40px
Label w:100px h:56px
Badge w:70px h:32px
Chip w:80px h:40px

These boxes sit inline (side by side, wrapping) but have explicit width and height. That's inline-block.

Classic use cases for inline-block: navigation links with padding, tag/badge chips, icon + text combos. In modern CSS, flex and grid handle most of what inline-block used to do — but knowing it matters for maintaining older codebases and understanding flow.

display: none

Element A — visible
Element B — display: none (you can't see me, and I take up no space)
Element C — visible (notice how it sits right after A, as if B doesn't exist)

display: none removes the element from the document flow entirely — it takes up no space and is invisible. This is different from visibility: hidden, which hides the element but preserves its space.

⚠️ Accessibility note: display: none also hides content from screen readers. Use aria-hidden="true" carefully, and consider visually-hidden CSS classes when content should be accessible but not visible.

2 — The position Property

The offset properties: top, right, bottom, left

Once an element has any position value other than static, you can nudge it using the four offset properties. These tell the element how far from each edge to move — but exactly what "edge" means differs by position value. That's what this section explores.

position: static — The Default

Static A position: static (default)
Static B top: 30px; left: 30px; — ignored ❌
Static C No offset — stays in normal flow

position: static is the default for every element. A static element is "not positioned" — offset properties (top, right, bottom, left) are completely ignored. z-index is also ignored on static elements.

position: relative

Box A — normal flow
Box B ghost (original space kept)
Box B — position: relative; top: -10px; left: 20px; Moved, but ghost space remains in flow
Box C — normal flow

position: relative does two important things:
1. Offsets the element visually, but its original space is preserved in the document flow (unlike absolute).
2. Creates a containing block — any position: absolute child inside it will position itself relative to THIS element, not the page. This second use is far more common than the first.

position: absolute

Example 1 — Absolute inside a Relative parent

position: relative (parent)
top:0; left:0
top:0; right:0
bottom:0; left:0
bottom:0; right:0
Centred top:50%; left:50%; transform:translate(-50%,-50%)

Common uses for position: fixed: navigation bars, floating action buttons (FABs), cookie banners, "back to top" buttons, chat widgets.

⚠️ transform, filter, or will-change on an ancestor element breaks position: fixed — it will position relative to that ancestor instead of the viewport.

position: sticky behaves like relative until the element reaches its offset threshold (e.g., top: 0), then it behaves like fixed within its scroll container.

⚠️ Two common gotchas:
1. You must set at least one offset (top, left, etc.) or it won't stick.
2. The parent must be scrollable AND must be taller than the sticky child. If the parent is overflow: hidden, sticky breaks.

3 — z-index & Stacking Order

z-index — Front-to-Back Stacking

z-index: 1
z-index: 2
z-index: 3 (front)
z-index: -1 (behind all)

z-index only works on positioned elements (anything with position other than static). Higher values appear in front. Negative values go behind. Without z-index, positioned elements stack in DOM order — later elements in the HTML are in front.

⚠️ Stacking Contexts — Why z-index Sometimes Breaks

Context A (z-index: 1 on parent)
child z-index: 999 Trapped inside Context A
Context B (z-index: 2 on parent)
child z-index: 1 Wins over Context A despite lower z-index

A stacking context is a self-contained z-index universe. Elements inside it are sorted among themselves, but the entire context competes as a unit with other contexts.

Things that create a new stacking context (the list is long, but common ones): position + z-index (not auto), opacity less than 1, transform, filter, will-change, isolation: isolate.

This is why a modal with z-index: 9999 can still appear behind a tooltip with z-index: 10 — if the modal lives in a lower stacking context.

🗂 Practical Z-Index Scale — UI Layer System

z-index: -1 Background decorations SVG patterns, pseudo-element shapes
z-index: 0 Default stacked elements Cards, content, regular positioned elements
z-index: 10 Dropdowns / Tooltips Select menus, hover tooltips
z-index: 100 Sticky headers / Sidebars Nav bars, fixed sidebars
z-index: 1000 Modals / Dialogs Overlay panels, lightboxes
z-index: 2000 Modal overlays (backdrop) Dark scrim behind modals
z-index: 9999 Notifications / Toasts Alert banners, top-layer elements

Use a defined z-index scale — never random numbers like z-index: 47 or z-index: 99999. Store your scale as CSS custom properties in :root (e.g., --z-modal: 1000). This prevents "z-index wars" where developers keep incrementing values to try to win against each other's elements.
