1 — Media Queries

What Are Media Queries?

A media query is a conditional CSS block — "apply these rules only when this condition is true." The most common condition is viewport width. Resize your browser window to see the demos below respond in real time.

Mobile-first means writing base styles for small screens, then using min-width queries to add complexity as the screen gets larger. This is the recommended approach because:
1. It forces you to prioritise content over decoration.
2. Small screens parse less CSS (no media query overhead).
3. You add complexity, never remove it — easier to maintain.

Desktop-First: @media (max-width: …)

Desktop layout (default)
Tablet override (max-width: 900px)
Mobile override (max-width: 600px)

max-width queries apply styles when the viewport is at most that wide. This is desktop-first — you write the large-screen layout as the default, then override it for smaller screens.

⚠️ Desktop-first is harder to maintain because you're subtracting features as screens shrink. Prefer mobile-first (min-width) for new projects. You'll still use max-width for targeted small-screen fixes on existing desktop-first codebases.

Modern Range Syntax — @media (600px <= width < 900px)

width < 600px
600px ≤ width < 900px
900px ≤ width < 1200px
width ≥ 1200px

CSS Media Queries Level 4 introduced a cleaner range syntax. Instead of @media (min-width: 600px) and (max-width: 899px), you can write @media (600px <= width < 900px).

Browser support: all modern browsers. The old syntax still works everywhere — use whichever reads more clearly to your team.

Media queries can test far more than width. Critical ones:
prefers-color-scheme: dark — respects OS dark mode.
prefers-reduced-motion: reduce — essential accessibility. Always wrap animations in a check for this. Users with vestibular disorders can be harmed by excessive motion.
hover: hover — touch devices have hover: none. Don't rely on hover-only interactions for critical UI.

2 — Container Queries

The Problem Media Queries Can't Solve

Imagine a card component. On a full-width page it has a horizontal layout. When placed in a sidebar, it should switch to vertical — but the viewport width hasn't changed. A media query can't know the card is in a sidebar. A container query can — it watches the card's parent container instead.

The card component has zero knowledge of where it lives. The parent declares itself a container, and the card's CSS adapts to the container's measured width — not the viewport.

Two required steps:
1. On the parent: container-type: inline-size (and optionally container-name: card-container).
2. On the component: @container (min-width: 400px) { … }

Browser support: all modern browsers since 2023. Safe for production.

Container Query Units: cqi, cqw, cqb

Container width drives font size via cqi

This text is 5cqi — 5% of the container's inline size.

This text is 3cqi — also relative to container width.

Container query units are like vw/vh but relative to the nearest named container instead of the viewport.
cqi = 1% of the container's inline size (usually width).
cqb = 1% of the container's block size (usually height).
cqw / cqh = 1% of container width / height explicitly.
Ideal for headings inside cards that need to scale with the card, not the page.

3 — Fluid Sizing with clamp()

clamp(minimum, preferred, maximum)

clamp() constrains a value between a minimum and maximum, with a preferred fluid value in between. It's the cleanest way to create type and spacing that scales continuously — no breakpoints needed. The preferred value is usually a viewport unit like 4vw.

Fluid Typography — Resize the browser to see scaling

Display Heading

font-size: clamp(2rem, 5vw, 4rem)

Section Heading

font-size: clamp(1.4rem, 3vw, 2.2rem)

Subheading

font-size: clamp(1.1rem, 2vw, 1.5rem)

Body text scales too. This paragraph uses font-size: clamp(0.9rem, 1.2vw, 1.1rem) — it's never smaller than 0.9rem or larger than 1.1rem, flowing smoothly between those extremes as the viewport changes width.

Without clamp(), type only changes at hard breakpoints — you'd need a media query for every font-size change. With clamp(), type scales continuously.

Reading the formula:
clamp(2rem, 5vw, 4rem)
— At viewport ≤ 640px: 5vw = 32px → min kicks in → stays 2rem (32px) ✅
— At viewport = 800px: 5vw = 40px → preferred value used
— At viewport ≥ 1280px: 5vw = 64px → max kicks in → stays 4rem (64px) ✅

Fluid Spacing — clamp() on padding and gap

Fluid padding

padding: clamp(1rem, 3vw, 3rem)

This card's padding grows with the viewport — tight on mobile, generous on desktop.

Fluid padding

padding: clamp(1rem, 3vw, 3rem)

Same component. Same rule. No breakpoints needed for spacing transitions.

clamp() works on any length property — font-size, padding, margin, gap, width, border-radius. It pairs perfectly with the rem + vw unit combination for accessible fluid scaling.

Siblings: min() and max()

width: min(100%, 600px)

Takes the SMALLER of 100% and 600px — fluid but capped. Better than max-width + width: 100% in one step.

width: max(300px, 50%)

Takes the LARGER of 300px and 50% — never smaller than 300px. A responsive min-width in one step.

clamp(a, b, c) is actually equivalent to max(a, min(b, c)) — max enforces the floor, min enforces the ceiling. Knowing this helps debug unexpected clamp behaviour.

4 — object-fit & object-position

All object-fit Values — Same image, same 240×160 box

fill (default)

Stretches to fill — distorts aspect ratio ⚠️

contain

Scales to fit — letterboxed, no crop

cover ✅ Most used

Fills box, crops edges — preserves ratio

none

Natural size — may overflow the box

scale-down

Smaller of none/contain — won't upscale

object-fit only works on replaced elements: <img>, <video>, <iframe>, <canvas>. The element must have an explicit width and height (or be inside a sized container) for object-fit to have anything to fit into.

cover is the workhorse — use it for hero images, card thumbnails, avatars, and any image container with a fixed aspect ratio.

object-position — Control the Crop Point

object-position: top left
object-position: center (default)
object-position: bottom right
object-position: 30% 20%

object-position works just like background-position — keywords (top, center, bottom, left, right) or percentage/pixel values. Use it to keep the focal point of an image in view after cropping — e.g., object-position: top center keeps faces visible in portrait photos used as landscape thumbnails.

6 — Breakpoint Quick Reference

Common Breakpoint Systems

Name Min-width value Target devices Common use
xs 0 (base) Small phones Base mobile styles
sm 480px / 30rem Large phones 2-col card grid
md 768px / 48rem Tablets Sidebar appears, nav expands
lg 1024px / 64rem Laptops 3-col layout, wider containers
xl 1280px / 80rem Desktops Max content width, 4-col grids
2xl 1536px / 96rem Large monitors Wider max-width, extra padding

⚠️ Don't treat breakpoints as sacred. The best breakpoints are where your content breaks — not where a particular device happens to be. Design your layout, then add breakpoints where things start looking wrong.

Using rem-based breakpoints is preferred — if the user increases their base font size, the breakpoint triggers sooner (when the layout starts to feel cramped), which is the correct behaviour.
