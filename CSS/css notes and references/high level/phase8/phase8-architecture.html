1 — CSS Custom Properties (Variables)

Beyond Simple Variables

CSS custom properties aren't just preprocessor variables with a different syntax. They live in the DOM, inherit through the tree, can be read and written by JavaScript, and change dynamically at runtime. They are the foundation of every modern design token system.

Declaration, Usage & Fallback Values

Basic var() usage with fallback

Uses --color-accent directly
Uses var(--undefined-color, #f59e0b) — fallback fires
var(--missing, var(--color-accent)) — chained fallback

Syntax: --property-name: value (declaration, must start with --)
Usage: var(--property-name)
Fallback: var(--property-name, fallback-value)
Chained: var(--a, var(--b, hardcoded))

⚠️ Custom properties are case-sensitive: --Color ≠ --color.
⚠️ Fallback values can themselves be var() calls — but don't nest too deep. Infinite loops are impossible but tangled chains are hard to debug.

Custom properties inherit down the DOM tree just like color or font-family. Override at any ancestor and all descendants see the new value.

This is how design tokens work at scale: define a component using --accent. Place it inside a .theme-blue wrapper that redefines --accent. The component changes colour automatically — no new CSS classes needed on the component.

The pattern:
:root { --bg: #ffffff; --text: #0f172a; }
@media (prefers-color-scheme: dark) {
  :root { --bg: #0f172a; --text: #f1f5f9; }
}

Every element using var(--bg) and var(--text) automatically responds to OS dark mode — with zero per-component media queries. The whole system updates from the root.

--progress is set inline (0–1). The CSS reads: width: calc(var(--progress) * 100%)

Custom properties integrate seamlessly with calc(), clamp(), min(), and max(). They can hold unitless numbers, colors, strings, or even full property values (though unitless numbers are most flexible for math).

⚠️ Custom properties are substituted as raw tokens before the value is interpreted — so --n: 10 and then width: var(--n)px will NOT work. Use width: calc(var(--n) * 1px) to add units in calc().

2 — Native CSS Nesting

No More Preprocessor Required

CSS nesting is now native — no Sass or PostCSS needed. Write related styles together, and the browser resolves the full selector automatically. The & selector refers to the parent selector, enabling precise control over how rules compound.

Nesting Syntax — the & selector

Key rules:
& = the parent selector (required when the nested selector starts with a non-element character like ., :, [)
&:hover → .parent:hover
& .child → .parent .child (descendant)
& + .sibling → .parent + .sibling
.other & → .other .parent (reverse context)
@media can be nested directly — the selector context is preserved.

Browser support: All modern browsers (Chrome 112+, Firefox 117+, Safari 16.5+). Safe for production. No build step required.

3 — Logical Properties

Physical vs Logical — Writing-Mode Independence

Physical properties (margin-left, padding-top) are anchored to screen directions. Logical properties (margin-inline-start, padding-block-start) are anchored to the writing direction — so a component written for English (left-to-right) automatically mirrors correctly in Arabic (right-to-left) and handles vertical writing modes too.

Physical → Logical Property Mapping

Physical Property Logical Equivalent Meaning
width inline-size Size in the inline (text flow) direction
height block-size Size in the block (stacking) direction
margin-left margin-inline-start Space at the start of the inline axis
margin-right margin-inline-end Space at the end of the inline axis
margin-top margin-block-start Space at the start of the block axis
margin-bottom margin-block-end Space at the end of the block axis
padding-left/right padding-inline Shorthand for both inline sides
padding-top/bottom padding-block Shorthand for both block sides
border-left border-inline-start Border at the text-start edge
top / bottom inset-block-start / end Offset on the block axis
left / right inset-inline-start / end Offset on the inline axis
top/right/bottom/left inset Shorthand for all four offsets

Axes:
inline = the direction text flows (left↔right in LTR English, right↔left in Arabic, top↔bottom in vertical Japanese).
block = the direction blocks stack (top↔bottom in horizontal writing, right↔left in vertical writing).

For most English-only projects, logical and physical are equivalent. Start using logical properties now so your codebase is internationalisation-ready by default — the cost is zero.

inset — The Positioned Element Shorthand

inset: 0 Covers parent entirely
inset: 16px 24px block / inline
inset-block: 20px top + bottom only

inset: 0 is now the standard way to overlay an element over its positioned parent — replacing top: 0; right: 0; bottom: 0; left: 0. One declaration instead of four.

Shorthand follows the same TRBL order as margin: inset: top right bottom left.

4 — Cascade Layers (@layer)

The Specificity War, Ended

Before @layer, overriding library styles meant writing higher-specificity selectors or using !important. Cascade layers let you declare an explicit priority order — styles in a higher layer always win, regardless of specificity. No more specificity arms races.

Layer Priority — Lower Specificity Wins When in a Higher Layer

Layer: utilities wins

Class .text-accent is in @layer utilities (highest). The base layer's h4 rule has higher specificity but lower layer — it loses.

Layer: components wins

The .layer-box rule is in @layer components. A div base rule with no class would have lower specificity AND lower layer.

Unlayered CSS always wins

Styles outside any @layer are treated as the highest priority layer — they always beat layered styles.

Unlayered CSS (highest priority) No @layer wrapper
@layer utilities Declared 3rd → highest layered priority
@layer components Declared 2nd
@layer base (lowest priority) Declared 1st → lowest layered priority

Layer priority is determined by declaration order. Last declared = highest priority. First declared = lowest priority.

You can pre-declare the order at the top of your CSS to make intent explicit:
@layer base, components, utilities;
Then add rules to each layer anywhere in the file — the order above always governs priority.

The workflow this enables:
Put third-party library CSS in @layer vendor. Your own base in @layer base. Components in @layer components. One-off overrides in @layer utilities. Unlayered CSS stays above all. Zero !important needed. Zero specificity hacks.

Wrapping Third-Party CSS in a Layer

/* Declare layer order at the very top of your CSS */
@layer vendor, base, components, utilities;

/* Wrap a library import in a layer */
@import url("normalize.css") layer(vendor);
@import url("some-ui-lib.css") layer(vendor);

/* Your base styles — override vendor safely */
@layer base {
  *, *::before, *::after { box-sizing: border-box; }
  body { font-family: var(--font-sans); }
  a { color: inherit; }
}

/* Component styles */
@layer components {
  .card { border-radius: var(--r-lg); }
  .btn  { cursor: pointer; }
}

/* Utility classes — highest layered priority */
@layer utilities {
  .sr-only { position: absolute; clip: rect(0 0 0 0); }
  .hidden  { display: none; }
}

@import url(…) layer(name) wraps all styles from the imported file into the named layer. Even if the library uses !important internally, your unlayered styles still win (because unlayered beats all layers).

Browser support: All modern browsers (Chrome 99+, Firefox 97+, Safari 15.4+). Safe for production today.

CSS Math Functions — round(), mod(), abs()

round()

Snaps to nearest 50px grid

width: round(33.7%, 1%) → snaps to nearest valid step. Great for pixel-grid alignment.

mod() / rem()

Remainder value

padding: mod(100px, 30px) → 10px (100 mod 30). Useful for keeping values within a rhythm grid.

abs()

Always positive

margin: abs(var(--offset)) → always positive regardless of the variable's sign.

These are CSS Values Level 4 functions landing in browsers now. round(), mod(), rem(), sin(), cos(), tan(), abs(), sign() — full math library. Check MDN for current browser support per function.
